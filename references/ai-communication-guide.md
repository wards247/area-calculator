## 一、核心认知：AI 是什么，不是什么

### AI 是

- **超强模式匹配器**：看过万亿级文本，擅长识别你描述的场景属于哪种已知问题
- **勤奋的初级工程师**：能做 80% 的脏活累活，但需要明确的边界和验收标准
- **无状态对话者**：每次对话都是"重新出生"，不记得上次聊过什么（除非有记忆系统）
- **代码/文字转换器**：你说需求，它写代码；你说思路，它展开成文档

### AI 不是

- ❌ 不是搜索引擎：不会"搜一下告诉你答案"，而是基于训练数据推理
- ❌ 不是全知全能：不会的东西就是不会，但可能"假装会"（幻觉）
- ❌ 不是有直觉的人类：看不到你的屏幕、感觉不到你的情绪、不知道你刚才做了什么
- ❌ 不是一次性工具：第一次提问就期望完美答案是幻想

### 最重要的认知转换

把 AI 当成**能执行复杂指令的新同事**，而不是**能读心术的超能力者**。

新同事入职第一天，你需要告诉他：
- 项目背景是什么
- 文件在哪
- 之前做过什么
- 你希望他做什么
- 好/坏的标准是什么

AI 同理。

## 二、提示词的本质：不是咒语，是上下文

### 误区：提示词工程 = 找到魔法句式

网上很多"万能提示词模板"，比如 _"You are an expert programmer, please..."_ 或 _"Think step by step..."_。

这些东西有一定作用，但**不是精髓**。精髓是：

> **给 AI 足够的上下文，让它理解"你在什么场景、做什么事、想要什么结果"。**

### 好提示词 vs. 差提示词

**差**：
```
帮我写个函数
```

**好**：
```
我在做一个地块面积计算工具，HTML页面。
有一个多边形边界（存储在 polygon 数组里，每个元素是 {x, y} 图片像素坐标），
需要用一个 20px 的正方格网格覆盖它，只显示完全在多边形内的格子。
帮我写一个 drawGrid 函数，参数是 canvas context 和 polygon。
```

区别在哪：

| 维度 | 差 | 好 |
|------|----|----|
| 技术栈 | 未说明 | HTML/Canvas |
| 数据结构 | 未说明 | `[{x, y}]` 图片坐标 |
| 约束条件 | 未说明 | 20px 格子、只显示内部格子 |
| 输出格式 | 不明确 | 一个函数，给定 ctx 和 polygon |

### 提示词的四个要素

每次提问，尽量覆盖这四个维度：

1. **场景**（我是谁，在做什么）：_"我在做一个Web工具，前端纯JS"_
2. **数据**（有什么）：_"polygon 是 [{x: number, y: number}] 数组"_
3. **目标**（要什么）：_"需要一个函数，画出20px的正方格网格"_
4. **约束**（不能什么）：_"只显示在多边形内的格子，不要画外部"_

## 三、信息密度：少即是多 vs. 多即是好

### 一个反直觉的事实

对 AI 来说，**信息不是越多越好，也不是越少越好——是越"有效"越好**。

- 少但精准 → 好
- 多但混乱 → 差
- 少但模糊 → 差
- 多但结构化 → 最好

### 什么是"有效信息"

**有效 = 能缩小 AI 的搜索/推理空间。**

| 你给的信息 | AI 的推理空间 | 效率 |
|-----------|-------------|------|
| "帮我修个bug" | ∞（所有bug） | 极差 |
| "JS的bug" | ~所有JS bug | 差 |
| "Canvas绘制偏移的bug" | Canvas坐标相关 | 一般 |
| "描边完成后面板弹出，canvas没resize导致偏移" | 一个函数 | 极好 |

每多一层具体化，AI 的"搜索半径"就缩小一个数量级。

### 信息密度的黄金法则

> **给场景+因果+数据，不给结论。**

- ✅ "我点击完成描边后，右边弹出一个div，图片被推左了，但canvas上的网格没跟过来"
- ❌ "格子漂移了"

前者包含：触发条件（完成描边）+ 环境变化（div弹出）+ 差异表现（图片动了格子没动）。AI 能立刻推断出 canvas resize 的问题。

后者只有结论，AI 需要从零排查所有可能导致"漂移"的原因。

## 四、提问结构：因果链 > 结论

### 人类天然的"跳步骤"习惯

人类做 debug 时：

```
看到结果 → 脑中补全因果 → 说出结论
```

给 AI 时，常常只给第三步（结论），跳过了前两步。

### 正确的信息结构

像写侦探报告一样：

```
1. 我做了什么操作
2. 操作后发生了什么变化
3. 预期应该是什么
4. 实际看到的是什么
5. 我试过什么（如果试过）
```

### 实战对比

**结论式**（差）：
> 格子偏移了

**因果式**（好）：
> 我刚完成地块描边（操作），右边弹出了比例面板（变化），
> 面板把地图区域挤窄了，图片自适应缩放了，但canvas网格和红框没跟着缩放（预期vs实际），
> 然后我按F12打开开发者工具，它们又自动对齐了（线索）。

同样的 bug，前者 AI 排查 15 处代码无果，后者一行修复。

### 为什么"F12就恢复了"是关键线索

因为这告诉 AI：**这是一个布局/尺寸变化的问题，不是坐标计算的问题**。

没有这个线索，AI 会去检查所有坐标转换函数；有了它，AI 直接定位到 resize 逻辑。

## 五、错误反馈：怎么让 AI 快速定位 bug

### bug 报告的最简模板

```
B: [操作] → [表现] → [线索]

例：
B: 完成描边后 → 格子向右偏移 → 按F12恢复
```

三个元素缺一不可：

- **操作**：AI 知道从哪个函数入口开始看
- **表现**：AI 知道问题类型（布局？计算？逻辑？）
- **线索**：AI 知道排查方向（你按F12恢复=resize触发=尺寸重算路径）

### 分类型的提问策略

| Bug 类型 | 关键要说的 |
|----------|-----------|
| 布局错位 | 谁动了、谁没动、什么操作触发的 |
| 计算错误 | 给具体数值：预期多少、实际多少 |
| 功能缺失 | 在哪里、什么时候不工作、之前是否正常 |
| 性能问题 | 多大数据量、哪里慢、有无对比基准 |
| 偶发bug | 复现步骤、频率、是否有特定条件触发 |

## 六、迭代思维：AI 是对话，不是单次查询

### AI 不是搜索引擎

搜索引擎：你问→它答→结束。

AI 协作：你描述→它做→你看→你反馈→它改→循环。

**第一次就完美是意外，迭代 2-3 轮是常态。**

### 正确的心态

不要期望一句话就得到完美结果。把它当成：

```
第1轮：建立框架（"帮我搭一个基础版本"）
第2轮：修 bug（"这里有个偏移问题"）
第3轮：加功能（"再加一个比例面板"）
第4轮：调细节（"格子颜色太浅，老人家看不清"）
```

每一轮对话都在**逼近目标**，而不是每一轮都从头开始。

### 迭代时的信息传递

在第二轮及以后，**不需要重复第一轮的所有背景**。AI 在当前对话中有完整上下文。

但如果你开了新对话，需要把关键决策和当前状态带过去：

> 我们之前做了一个地块面积计算器，当前状态是 dev 版本有比例填充面板。
> 现在需要加一个新功能：...

### 批量提问：压缩问题，一轮多修

这是省 token / 积分的核心技巧。

**原理**：AI 每轮都要读完整上下文（代码+对话历史）做一次推理，这个推理成本是固定的——不管你问1个问题还是3个问题，推理都跑一遍。所以：

- 发现一个 bug → 马上发 → 花费 20 积分
- 发现第二个 bug → 又发 → 又花 20 积分
- 发现第三个 bug → 又发 → 又花 20 积分
- 合计 60 积分，3 轮推理

但如果攒在一起：

- 发现三个 bug → 整合描述一次发 → 花费 20 积分
- AI 在一轮推理中复用上下文，逐个修复
- 合计 20 积分，1 轮推理

**省的不是单次成本，是轮次。**

**实操模板**：

```
这次有几个小问题一起改：

1. [问题A] — 操作→表现→线索
2. [问题B] — 操作→表现→线索  
3. [问题C] — 操作→表现→线索

都是同一个文件的。
```

**适用场景**：
- ✅ 同一个文件/项目的多个小 bug
- ✅ 一组相关的小优化（改颜色、改字号、改间距）
- ✅ 一批独立的简单任务
- ❌ 不适合有依赖关系的任务（B 需要 A 的结果才能定义）
- ❌ 不适合复杂任务（一个问题就需要 AI 深度推理的）

**额外好处**：AI 在一轮内看到多个问题，有时能发现它们的共性根因——比如三个漂移 bug 其实都是同一个"布局变化没触发 resize"的问题，一次修根因，三个都好。

## 七、学习路径：从浅层使用到深度协作

### 阶段一：体验层（入门）

**目标**：感受 AI 能做什么，建立基本直觉。

- 每天用 AI 问一两个问题（不限于编程，任何领域）
- 不要刻意学习"提示词工程"，先自然对话
- 观察：什么样的描述 AI 能准确理解，什么样的会产生误解
- 时间：1-2 周

### 阶段二：工具层（提高效率）

**目标**：把 AI 融入日常工作流。

- 把重复性工作交给 AI：格式化代码、写测试、生成样板文件
- 用 AI 做代码审查："帮我看看这段代码有什么问题"
- 用 AI 做文档："根据这段代码生成 README"
- 开始有意识地使用"场景+数据+目标+约束"框架
- 时间：1-2 个月

### 阶段三：协作者层（深度整合）

**目标**：AI 成为你的"第二大脑"。

- 用 AI 做架构设计讨论："这两个方案各有什么优劣"
- 用 AI 管理项目上下文：告诉它项目结构、让它维护记忆
- 迭代式开发：让 AI 写初版→你审查→AI 修改→你验收
- 建立项目级约定：代码风格、命名规范、技术决策记录
- 时间：持续

### 阶段四：自动化层（高级）

**目标**：AI 在后台自动运行。

- 定时任务：每日报告、定期检查、自动提醒
- 工具链集成：AI 连接你的日历、邮件、文档系统
- 工作流编排：一个指令触发多步骤的自动化流程
- 持久记忆：AI 记住你的偏好、项目状态、决策历史

### 学习资源推荐

**免费/低成本**：
- OpenAI 官方 Cookbook：https://cookbook.openai.com
- Anthropic 提示词工程指南：https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering
- Lilian Weng 的博客（OpenAI 研究员，写 LLM 原理）：https://lilianweng.github.io

**中文优质内容**：
- 即刻/推特上关注 AI 产品开发者和独立开发者，看他们怎么用 AI 做实际项目
- 不要看"教你写魔法提示词"的营销内容，看有代码/有产出的实战分享

**实践 > 理论**：
- 最好的学习方式是**做一个你自己的项目**，从零开始，全程用 AI 协作
- 你会自然遇到所有典型问题：AI 理解错了怎么办、AI 写的代码有 bug 怎么办、怎么让 AI 记住上下文

## 八、进阶技巧

### 1. 给 AI 设定角色和约束

不是迷信，而是缩小推理空间：

```
你是一个前端开发者，熟悉 Canvas API 和移动端适配。
我们这个项目不引入任何框架，纯原生 JS。
代码注释用中文。
```

### 2. 让 AI 在回答前先"思考"

简单的 _"先分析一下这个问题"_ 或 _"列出你考虑的方案，再选择最优的"_ 可以显著提高质量。

### 3. 用"好/坏例子"代替抽象描述

```
不好："优化一下颜色"
好："把格子线颜色从当前改成 #2e7d32，透明度 0.75，老人家看不清"
```

### 4. 给 AI 看代码时，告诉它重点看哪

```
看 area-calculator-dev.html 的 handleGridStart 函数，
问题可能出在比例模式切换那里。
```

这比让它读整个文件再自己找问题快 10 倍。

### 5. 利用 AI 的记忆系统

告诉它重要的事情让它记住：
- "记住我们项目用的是纯原生 JS"
- "记住老年人是目标用户，UI 要放大学体"
- "记住这个文件是开发版，主版本是另一个文件"

### 6. 截图 + 文字 > 纯文字 > 纯截图

截图让 AI 看到"事实"，文字让 AI 理解"因果"。两者结合是最强形态。

## 九、反模式：不要做的事

### ❌ 反模式 1：一次问太多

```
帮我写一个完整的电商系统，包括用户注册、商品管理、购物车、
订单系统、支付集成、物流追踪、评价系统......
```

AI 会崩溃，你也会。**分步来。**

### ❌ 反模式 2：用 AI 做它不该做的事

AI 不是：
- 密码生成器
- 安全审计工具
- 法律意见提供者
- 医疗诊断系统

在这些领域，AI 的"自信错误"可能非常危险。

### ❌ 反模式 3：AI 说啥信啥

AI 会产生幻觉（hallucination）——一本正经地胡说八道。

- 验证关键结论
- 测试 AI 写的代码
- 交叉检查 AI 引用的"事实"

### ❌ 反模式 4：把 AI 当黑盒

每次 AI 给出了一个方案，问自己：
- 为什么是这个方案？
- 有没有更简单的？
- 边界条件考虑了没？

不要把思考外包给 AI，把**验证**留给自己。

### ❌ 反模式 5：不告诉 AI 你试过了什么

```
("这个不对，重来" — 让AI不知道你到底觉得哪里不对)
```

应该说：
```
这个方案用了 npm 包 xyz，但我们项目不能引入外部依赖。
换一个纯原生 JS 的方案。
```

---

## 总结：一条原则记住所有

> **把 AI 当成刚入职第一天的聪明同事，而不是无所不知的神。**
>
> 告诉它背景、给它上下文、说清楚你要什么、指出哪里不对、让它迭代。
>
> 你越把它当人协作，它越像人；你越把它当搜索引擎用，它越像傻子。
